# 我在 Windows 下写了个 AI 调度器，然后被编码地狱暴打了一下午

> 零感团队 · 2026年6月 · 阅读约 8 分钟

---

## 前提：一切都很美好

AIMate（零感调度器）的代码在 macOS 上写的，Python + FastAPI + Jinja2，干净利落。逻辑跑通、API 调通、页面渲染——一切正常。我心里想的是：这玩意装到 Windows 上双击就能跑。

然后我在 Windows CMD 里跑了 `python server.py`。

---

## 第一关：CMD 乱码

```
'ython' 不是内部或外部命令，也不是可运行的程序或批处理文件。
```

`python` 被吃掉了第一个字符 `p`。终端里满是方框、问号和残破的中文。

原因是 Windows CMD 的默认代码页是 936 (GBK)。我的 `start.bat` 文件虽然写了 `chcp 65001` 切换到 UTF-8，但文件本身的换行符是 LF——CMD 解析 LF 换行的批处理文件时行为不可预测，`chcp` 命令有时根本没被执行到。

**解决：用 `printf` 生成 CRLF 换行的 bat 文件，内容全部用 ASCII 字符（不含中文注释），确保 CMD 解析不出错。**

教训：Windows 批处理文件必须是 CRLF，而且第一行 `@echo off` 和 `chcp 65001` 之间不能有任何非 ASCII 字符。

---

## 第二关：`/dev/null` 消失术

```bash
# start.bat 里有这行
pip install ... >/dev/null 2>&1
```

提交代码后，linter/pre-commit hook 自动把 `/dev/null` 改成了 `>nul 2>&1`。这个自动改写本身是对的——Windows 没有 `/dev/null`——但它把语法也改了，导致管道重定向断裂。

**解决：直接用 `>nul 2>nul`，绕开任何可能被自动改写的写法。**

教训：如果你在跨平台项目里写 shell 重定向，要么全部用平台无关写法，要么在 `.gitattributes` 里标记 bat 文件不走 linter。

---

## 第三关：Jinja2 的幽灵 Internal Server Error

服务器启动了，`curl http://127.0.0.1:7799/` 却返回 500 Internal Server Error。日志里没有任何有用信息。

花了 40 分钟排查：
- FastAPI 路由没问题
- 端口没冲突
- 文件路径正确

最后发现是 **Jinja2 3.1.6 的缓存键 bug**。我的 `index.html` 根本没用到任何 Jinja2 模板变量（纯静态 HTML），但 `Jinja2Templates` 仍在尝试解析和缓存它。在某些 Windows Python 环境下，模板缓存的路径规范化出了问题，导致内部异常。

更讽刺的是：页面里一个 `{{ }}` 都没有，根本不需要模板引擎。

**解决：删掉 Jinja2Templates，直接用 `(ROOT / "templates" / "index.html").read_text()` 返回 HTMLResponse。**

教训：别为"将来可能用到"引入依赖。如果 HTML 是纯静态的，就用文件读取，不要挂模板引擎。

---

## 第四关：端口幽灵进程

```bash
python server.py
# OSError: [Errno 10048] 端口 7799 已被占用
```

上次测试的 Python 进程没被杀干净，端口还被占着。`Ctrl+C` 只关了 uvicorn，没关 bridge.py 的子进程。

**解决：`start.bat` 第一件事——`netstat -ano | findstr :7799` 找到 PID → `taskkill /F /PID` 强杀，然后再启动。**

教训：Windows 进程管理不像 Unix 那么干净。SIGTERM 不一定生效，子进程不会自动跟随父进程退出。最佳实践是启动前先清端口。

---

## 修复汇总

| 问题 | 根因 | 修复 |
|------|------|------|
| CMD 乱码/命令被截断 | LF 换行 + GBK 代码页 | CRLF + ASCII + chcp 65001 |
| `/dev/null` 被改写 | Linter 自动"修正" | 用 `>nul 2>nul` |
| Jinja2 Internal Server Error | 缓存键 bug + 不需要模板引擎 | 去掉 Jinja2，直接读文件 |
| 端口占用 | 子进程残留 | 启动前自动杀旧进程 |

---

## 这件事教会我什么

**跨平台开发不是"在 macOS 上写，Windows 上测一下"。** 是"每个平台有自己的一套规则，你不想学也得学"。

具体到 Windows Python 开发：

1. **换行符真的会杀人。** bat 文件必须 CRLF，py 文件无所谓但保持一致。
2. **不要假设 Unix 工具存在。** `/dev/null`、`grep`、`awk` 在 Windows 上都可能不在 PATH 里。
3. **模板引擎是静态页面的过度工程。** 如果 HTML 不需要变量注入，别用模板引擎。
4. **进程生命周期需要显式管理。** 启动前杀旧进程，关闭时清理子进程。不能依赖 `Ctrl+C`。

但修好之后，双击 `start.bat`，浏览器自动打开，看到页面干干净净的那一刻——值了。
